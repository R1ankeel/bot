## 1. Что это такое

**НейроАнька** — чат-бот для платформы **galaxy.mobstudio.ru**. Бот работает через браузерную автоматизацию **Playwright**: читает сообщения из DOM, ведёт память в SQLite, анализирует общий чат, генерирует ответы через **xAI API / Grok** и отправляет сообщения обратно в чат.

Актуальная версия умеет:

- отвечать на обращения по имени и на команды;
- склеивать несколько сообщений пользователя в одну LLM-задачу, если ответ ещё не сгенерирован;
- хранить память о пользователях в виде фактов, ярлыков, истории и структурированного профиля;
- учитывать пол пользователя и подбирать правильный род обращения;
- вести гибридную историю диалога: свежие сообщения + первые «якорные» ходы;
- помнить открытый вопрос бота и корректно отвечать на короткие follow-up реплики;
- отслеживать эмоциональный тон пользователя;
- учитывать общий контекст чата и упомянутых людей;
- отвечать на сторонние упоминания самой Аньки в специальном SELF_IDENTITY-режиме;
- использовать web search для культурных и актуальных фактических запросов;
- намеренно **не искать** погоду и сложные школьно-научные темы, а отвечать как персонаж;
- защищаться от prompt injection;
- снижать «заражение» странной лексикой пользователя через карантин слов;
- избегать повторов через антиповтор, style session и блокировку паттернов;
- вести репутацию, уровни отношений и единственного любимчика;
- ревновать любимчика, реагировать на флирт соперников и иногда игнорировать сообщения в afterglow;
- выдавать достижения;
- вести дневник планеты и публиковать вердикт дня после 22:00;
- приветствовать входящих с вероятностью 25%;
- редко встревать в активный общий диалог коротким язвительным боковым комментарием;
- оживлять чат хуками при долгой тишине.

## 2. Требования и запуск

### Зависимости

```bash
pip install playwright httpx ddgs
playwright install chromium
```

`ddgs` может оставаться в зависимостях по историческим причинам, но актуальный поиск в коде идёт через xAI Responses API с инструментом `web_search`.

### Настройка API

Ключ и модели задаются в `config.py`:

```python
XAI_API_KEY    = "xai-..."
MODEL          = "grok-4-1-fast-reasoning"
MODEL_ANALYSIS = "grok-4-1-fast-reasoning"
```

**Важно:** не публикуй реальный `XAI_API_KEY` в репозитории и не вставляй его в документацию.

### Запуск

```bash
python main.py
```

После запуска откроется Chromium. Авторизация на портале выполняется вручную: бот сам не логинится.

## 3. Архитектура

Бот работает асинхронно через `asyncio`.

```text
main loop
  ↓
parse_messages(page)
  ↓
find_new_messages(...) + дедуп cache
  ↓
log_global_message(...) + state.buffer_add(...)
  ↓
команды / триггеры / ревность / действия
  ↓
state._reply_queue
  ↓
_reply_worker(page)
  ↓
responder / llm / formatter
  ↓
send_response(page, text)
```

### Главные принципы

- `main.py` только читает чат, распознаёт события и кладёт задачи в очередь.
- `worker.py` последовательно обрабатывает очередь и соблюдает кулдаун отправки.
- `responder.py` отвечает за LLM-генерацию и форматирование командных ответов.
- `database.py` хранит долговременную память.
- `state.py` хранит временное состояние процесса.
- `prompt_builder.py` собирает контекст для модели.

## 4. Основной цикл работы

`main.py` делает следующее:

1. Инициализирует БД через `init_db()`.
2. Добавляет системных игнорируемых пользователей через `seed_ignored(IGNORED_USERS)`.
3. Мигрирует старую JSON-историю, если есть `HISTORY_FILE`.
4. Запускает Chromium через Playwright.
5. Создаёт `state._reply_queue` и запускает `_reply_worker(page)`.
6. В цикле парсит сообщения из DOM.
7. Определяет новые сообщения через `find_new_messages(...)`.
8. Дедуплицирует сообщения через `processed_message_keys`.
9. Логирует сообщения в `global_chat`.
10. Добавляет сообщения в общий буфер `state._global_buffer`.
11. Проверяет автоответы, ревность, действия, команды и обращения по имени.
12. При активном мини-диалоге может поставить в очередь `group_comment` со свежим контекстом общего чата.
13. При необходимости запускает фоновые задачи фактов, дневника и репутации.
14. После 22:00 ставит в очередь `daily_verdict`, если за дату ещё нет отчёта.
15. При тишине в чате генерирует хук.

### Дедупликация DOM-сообщений

В `browser.py` для каждого сообщения строится стабильный `key` из:

- ника;
- видимого времени;
- нормализованного текста;
- счётчика дублей одинаковых сообщений.

Если DOM-якорь потерян, `find_new_messages(...)` возвращает все сообщения, а `processed_message_key_set` в `main.py` отсекает уже обработанные.

## 5. Очередь ответов

Очередь хранится в `state._reply_queue`.

### Склейка LLM-сообщений

`state.enqueue_llm_job(username, text, mood)` не всегда создаёт новую задачу. Если от этого пользователя уже есть ожидающий `llm` job, новый текст приклеивается к старому:

```text
первое сообщение
второе сообщение
третье сообщение
```

Текст ограничивается `MAX_MERGED_LLM_TEXT = 1200`, при переполнении сохраняется более свежая часть. Это решает проблему, когда человек пишет по одному слову, а бот отвечает на каждое слово отдельно.

### Типы задач в очереди

Актуальные типы задач:

| Тип | Откуда появляется | Что делает |
|---|---|---|
| `llm` | обращение по имени, действие с ботом, стороннее упоминание | обычный ответ через LLM |
| `text` | автоответы, ошибки команд, системные ответы | отправка готового текста |
| `search` | `!найди` | принудительный web search |
| `greeting` | вход пользователя на планету | приветствие |
| `group_comment` | активный мини-диалог в общем чате | короткая боковая реплика без обращения по нику |
| `comfort` | `!утешить` | саркастическое утешение |
| `roast` | `!заагрись` | агрессивный ответ/подкол |
| `opinion` | `!мнение` | мнение бота о пользователе |
| `compatibility` | `!совместимость` | оценка совместимости двух ников |
| `find_pair` | `!найдипару` | подбор пары из пользователей с памятью |
| `horoscope` | `!гороскоп знак` | персональный гороскоп по памяти |
| `profile_view` | `!профиль` / `!profile` | карточка профиля |
| `forget` | `!забудь` / `!forget` | сброс части памяти |
| `history_view` | `!история` / `!history` | краткая история общения |
| `achievements_view` | `!достижения` | список достижений |
| `daily_verdict` | ежедневная проверка после 22:00 | публичный вердикт дня |
| `jealousy` | ревность к любимчику | ревнивая реакция |
| `jealousy_rival` | флирт соперника рядом с любимчиком | ответ сопернику |

### Воркер

`worker.py` берёт задачи по одной. Для `llm` он:

1. удаляет задачу из `_pending_llm_jobs`;
2. извлекает необычные слова пользователя и кладёт их в карантин;
3. расходует один post-return strike ревности, если он активен;
4. вызывает `get_response(...)`;
5. отправляет ответ, если он не пустой;
6. проверяет новые достижения;
7. каждые `REPUTATION_EVAL_EVERY` сообщений запускает пересчёт репутации.

## 6. Модули проекта

### `main.py`

Точка входа. Управляет браузером, главным циклом, командами, дедупликацией, приветствиями, хуками, фоновыми задачами и постановкой задач в очередь.

Особенности:

- `!найдипару` имеет кулдаун 1 час на пользователя;
- `group_comment` имеет локальный кулдаун `GROUP_COMMENT_COOLDOWN = 180` секунд и ставится в очередь только при активном мини-диалоге;
- `!забудь` и `!история` доступны только админу;
- `!факты` в текущем коде доступна всем и показывает факты о самом вызвавшем пользователе;
- сторонние упоминания бота переводятся в SELF_IDENTITY-ввод;
- действия без ника обрабатываются как `[ДЕЙСТВИЕ]`.

### `worker.py`

Последовательно обрабатывает `_reply_queue`. Здесь находятся ветки обработки всех типов задач, включая `group_comment`, `find_pair`, `horoscope`, `jealousy_rival`, `daily_verdict`, `profile_view`, `forget`, `history_view`, `achievements_view`.

Для `group_comment` используется отдельная компактная генерация, не общий `get_system_prompt()`: воркер получает свежий кусок общего чата из job, определяет тон сцены и просит модель дать одну короткую боковую реплику без ника в начале.

### `responder.py`

Главный модуль генерации ответов.

Основные функции:

- `get_response(...)` — обычный ответ на обращение;
- `get_forced_search_response(...)` — `!найди`;
- `get_greeting(...)` — приветствие вошедшего;
- `get_self_opinion_response(...)` — `!мнение`;
- `get_compatibility_response(...)` — `!совместимость`;
- `get_find_pair_response(...)` — `!найдипару`;
- `get_horoscope_response(...)` — `!гороскоп`;
- `get_daily_diary_summary(...)` — `!дневник`;
- `format_user_profile_info(...)` — `!профиль`;
- `format_forget_response(...)` — `!забудь`;
- `format_history_summary(...)` — `!история`;
- `format_reputation_info(...)` — `!репутация`.

Важная постобработка ответа:

1. `ensure_nick(...)` — гарантирует ник в начале;
2. `fix_username_in_response(...)` — исправляет искажённый ник;
3. `deduplicate_username_in_response(...)` — убирает повтор ника внутри ответа;
4. `normalize_strict_nick_prefix(...)` — приводит начало к формату `Ник, текст`.

### `prompt_builder.py`

Собирает системный промпт из блоков:

- отношение и репутация;
- профиль пользователя;
- гендерный блок;
- факты и ярлыки;
- current-thread block;
- current relation mood;
- mentioned people block;
- recent chat block;
- ревнивые подсказки;
- anti-repeat/pattern block;
- style session;
- hidden reminders;
- текущее время и активность.

Промпт подрезается под токен-бюджет. Для русского используется грубая оценка `~3 символа = 1 токен`.

### `database.py`

SQLite-хранилище. Помимо CRUD содержит:

- миграции старых таблиц;
- профиль пользователя `user_profile`;
- realtime-обновление профиля;
- batch-обновление профиля из фактов;
- нормализацию городов;
- защиту профиля от prompt injection;
- дедупликацию конфликтующих фактов;
- гибридную историю;
- salience ranking фактов;
- локальные даты через `LOCAL_TZ_OFFSET_HOURS`.

### `analysis.py`

Фоновая аналитика:

- извлечение фактов в два этапа: кандидаты → валидация;
- извлечение ярлыков;
- разделение ярлыков на `self_declared` и `observed`;
- сохранение фактов и ярлыков;
- обновление структурированного профиля;
- очистка конфликтующих фактов;
- генерация дневника планеты;
- анализ репутации пользователя.

### `llm.py`

Совместимая обёртка под старый Ollama-интерфейс, но фактически работает через xAI API.

Функции:

- `ollama_chat(...)` — основной вызов Chat Completions;
- `ollama_analyze(...)` — аналитический вызов с пониженной температурой;
- `ollama_chat_with_web_search(...)` — Responses API с `tools=[{"type": "web_search"}]`;
- `unload_model(...)`, `preload_model(...)`, `restart_ollama()` — заглушки совместимости.

`ollama_chat(...)` делает до 3 попыток при `429`, `502`, `503`, `504` и сетевых ошибках.

### `browser.py`

Playwright I/O:

- парсинг обычных сообщений;
- парсинг действий без ника;
- извлечение смайлов платформы;
- построение стабильного ключа сообщения;
- поиск новых сообщений;
- отправка текста;
- разбиение длинных ответов по `CHUNK_SIZE` с задержкой `CHUNK_DELAY`.

### `utils.py`

Чистые утилиты:

- очистка ответа;
- проверка обращения по имени;
- prompt injection detector;
- классификация поиска;
- работа с фактами;
- защита от сырого дампа чата;
- SELF_IDENTITY-логика стороннего упоминания бота;
- карантин необычных слов;
- нормализация ника в ответе.

### `state.py`

Временное состояние процесса:

- очередь и pending LLM jobs;
- счётчики сообщений;
- таймеры фоновых задач;
- глобальный буфер чата;
- текущий режим;
- текущая активность;
- состояние ревности;
- pending dialog;
- emotion state;
- style session;
- pattern block state;
- vocab quarantine.

### `reputation.py`

Уровни отношений, прогресс, реакции на смену уровня, влияние усталости от спама и штрафы ревности.

### `jealousy.py`

Отслеживание ревности к текущему любимчику, реакции на соперников, afterglow, игнор сообщений любимчика и ревнивые подсказки в промпт.

### `achievements.py`

Каталог достижений, проверка условий и форматирование списка.

### `daily_verdict.py`

Генерация публичного вердикта дня после 22:00. Вердикт сохраняется в `daily_reports`, чтобы не отправляться повторно за ту же дату.

## 7. Память, профиль и контекст

### 7.1. Сырые факты

Факты хранятся в `user_facts`:

```text
username | fact | source | created_at
```

Фоновый анализ берёт необработанные сообщения из `global_chat`, извлекает кандидатов, валидирует их и сохраняет только подтверждённые факты. Дополнительно фильтруются факты, которые слишком похожи на дословную цитату из чата.

### 7.2. Ярлыки

Ярлыки хранятся в `user_traits` и делятся по источнику:

- `self_declared` — пользователь сам сказал о себе;
- `observed` — бот вывел по поведению;
- `chat` — старый/совместимый источник.

В промпте self-declared ярлыки имеют больший приоритет, чем observed.

### 7.3. Структурированный профиль

Таблица `user_profile` хранит канонические поля:

| Поле | Значение |
|---|---|
| `display_name` | как пользователь просит его называть |
| `real_name` | настоящее имя, если известно |
| `gender` | `male` / `female` / `unknown` |
| `age` | возраст |
| `city` | город |
| `interests` | JSON-список интересов |

Профиль обновляется двумя путями:

1. `update_profile_realtime(...)` — сразу из прямого сообщения;
2. `extract_and_update_profile_from_facts(...)` — из накопленных фактов после анализа.

Примеры realtime-обновлений:

| Сообщение | Поле |
|---|---|
| `называй меня Саша` | `display_name` |
| `меня зовут Саша` | `real_name` |
| `мне 23 года` | `age` |
| `я из Питера` | `city` |
| `я девушка` | `gender=female` |
| `я парень` | `gender=male` |

### 7.4. Защита профиля от инъекций

Значения профиля проверяются через `_is_safe_profile_value(...)`. Отклоняются слишком длинные значения, HTML/XML, скобки, подозрительные слова вроде `system`, `prompt`, `игнорируй`, `ты теперь`.

### 7.5. Identity block

`build_identity_block(username)` всегда добавляет в промпт компактный профиль пользователя: имя, пол, возраст, город, интересы. Это основная защита от забывания базовых данных.

### 7.6. Гендерный блок

`build_gender_block(username)` добавляет жёсткую инструкцию по роду:

- для девушки: `пришла`, `написала`, `сказала`, `рада`;
- для парня: `пришёл`, `написал`, `сказал`, `рад`.

Приоритет определения пола:

1. `user_profile.gender`;
2. `get_gender_fact(...)`;
3. эвристика по фактам и ярлыкам.

### 7.7. Гибридная история

`get_history_hybrid(username, recent_turns=8, anchor_turns=2)` передаёт модели:

- последние 8 ходов;
- первые 2 якорных хода знакомства.

Так бот помнит свежий контекст и не полностью теряет начало отношений.

### 7.8. Pending dialog

Если бот задал вопрос, `state.set_pending_dialog(...)` запоминает его на 5 минут. Следующая короткая реплика пользователя считается ответом на этот вопрос, а не новым разговором.

### 7.9. Emotion state

`update_user_emotion(...)` отслеживает тон последних сообщений пользователя:

- `flirty`;
- `angry`;
- `positive`;
- `negative`;
- `neutral`.

Этот тон попадает в current-thread block.

### 7.10. Recent chat context

`build_recent_chat_block(limit=8, query=user_message)` подтягивает сообщения из общего буфера. При наличии query выбираются более релевантные текущей теме сообщения.

### 7.11. Mentioned people context

Если пользователь спрашивает о другом участнике, `build_mentioned_people_block(...)` добавляет факты и ярлыки упомянутого человека. Для коротких ников используется проверка границ слова, чтобы не ловить подстроки внутри других слов.

### 7.12. SELF_IDENTITY-режим

Если бота упомянули как объект разговора, например:

```text
Маша, кто такая Анька?
Ваня, что думаешь про НейроАньку?
```

`utils.looks_like_third_party_bot_mention(...)` переводит сообщение в специальный структурированный prompt через `build_third_party_bot_mention_prompt(...)`.

В этом режиме responder:

- отвечает про саму Аньку от первого лица;
- не превращает адресата перед запятой в target_user;
- не подключает mentioned people block;
- не тянет random recent chat context.

## 9. Система поиска

### 9.1. Принудительный поиск `!найди`

`!найди запрос` кладёт в очередь задачу `search`. Воркер вызывает `get_forced_search_response(...)`, а та использует `ollama_chat_with_web_search(...)`.

Правила ответа:

- использовать web search;
- отвечать коротко и в стиле Аньки;
- не говорить «я нашла в интернете»;
- не вставлять URL;
- начинать строго с ника пользователя.

### 9.2. Автопоиск

`utils.get_search_kind(...)` классифицирует сообщение:

| Вид | Поведение |
|---|---|
| `culture` | использовать web search для песен, фильмов, игр, аниме, мемов |
| `factual` | использовать web search для новостей, дат, цен, расписаний, результатов |
| `blocked_weather` | не искать и не давать конкретный прогноз |
| `blocked_science` | не искать и не изображать эксперта |
| `""` | обычная генерация без поиска |

### 9.3. Культурные запросы

Для музыки, песен, фильмов, игр и мемов бот использует поиск, чтобы не выдумывать факты. Если просят «напой» песню, бот не должен выдавать фейковый официальный текст. Допустима короткая авторская стилизация и честное уточнение, что это не дословная цитата.

### 9.4. Погода и школьно-научные темы

Погода специально заблокирована от поиска. Бот не должен уверенно называть температуру, осадки или ветер. Он отвечает как персонаж: уклоняется, шутит, предлагает посмотреть прогноз самому.

Сложные научные/учебные темы тоже блокируются от поиска: бот не должен превращаться в энциклопедию, а отвечает как обычная девчонка из чата.

### 9.5. Почему это важно

Так решаются две противоположные проблемы:

- бот не выдумывает факты там, где подросток мог бы найти культурную информацию;
- бот не становится «погодным виджетом» или учебником там, где по характеру ему лучше отшутиться.

## 10. Система репутации

Репутация хранится в таблице `reputation`.

### Уровни

| Уровень | Название |
|---:|---|
| 0 | Любовь |
| 1 | Влюблённость |
| 2 | Влечение |
| 3 | Дружба |
| 4 | Привязанность |
| 5 | Приветливость |
| 6 | Нейтральность |
| 7 | Неприязнь |
| 8 | Вражда |
| 9 | Ненависть |

Стартовый уровень — **Нейтральность**.

### Прогресс

Для каждого уровня есть свой порог `LEVEL_PROGRESS_CAP`. Позитивные дельты двигают пользователя к лучшим уровням, негативные — к худшим.

### Анализ репутации

`state.REPUTATION_EVAL_EVERY = 10`: после каждых 10 прямых сообщений пользователя бот может запустить `evaluate_reputation_background(...)`.

LLM возвращает число от `-3` до `+3`:

- `+3` — сильная забота, флирт, поддержка;
- `+2` — заметный плюс;
- `+1` — слабый плюс;
- `0` — обычный разговор;
- `-1` — лёгкая грубость;
- `-2` — токсичность;
- `-3` — сильная агрессия/унижение.

Обычные вопросы про технику, игры, новости и советы считаются `0`, если в них нет грубости.

### Любимчик

Уровень `Любовь` может быть только у одного человека. Если новый пользователь доходит до `Любовь`, предыдущий любимчик понижается.

## 11. Система ревности

Ревность работает только к текущему любимчику (`is_lover = 1`).

### Что отслеживается

- любимчик пишет в общий чат, но не обращается к боту;
- любимчик явно флиртует или обсуждает поцелуи/нежности;
- другой пользователь флиртует рядом с любимчиком;
- любимчик возвращается к боту после периода игнора.

### Контекст

`jealousy.py` классифицирует сообщения по маркерам:

- обычное общение;
- слишком тёплое внимание;
- флирт;
- явный флирт;
- откровенный флирт.

Чем сильнее контекст, тем больше штраф и тем быстрее реакция.

### Стадии и стили

У ревности есть стадии и стили:

| Стиль | Смысл |
|---|---|
| `icy` | ледяной игнор |
| `passive` | пассивная агрессия |
| `possessive` | собственническая предъява |
| `meltdown` | публичная истерика |
| `rival_agg` | агрессия к сопернику |

### Afterglow

После вспышки ревности включается afterglow на 300–600 секунд. В это время бот может оставаться колким даже после прямого обращения любимчика.

### Игнор любимчика

В состоянии ревности бот может намеренно не ответить любимчику:

```python
JEALOUSY_IGNORE_CHANCE = 0.10
JEALOUSY_IGNORE_CHANCE_AFTERGLOW = 0.18
```

### Реакция на соперника

Если соперник флиртует с любимчиком, ставится задача `jealousy_rival`. Бот может ответить уже сопернику и обозначить, что любимчик — «мой парень», «моя девушка» или «мой человек» в зависимости от профиля.

### Основные настройки ревности

| Параметр | Значение | Описание |
|---|---:|---|
| `JEALOUSY_PUBLIC_MESSAGE_TRIGGER` | 6 | обычное общение до вспышки |
| `JEALOUSY_PUBLIC_MESSAGE_TRIGGER_SOFT` | 3 | ускоренный порог для флирта |
| `JEALOUSY_INSTANT_PENALTY_THRESHOLD` | 4 | сильный флирт → почти мгновенная реакция |
| `JEALOUSY_COOLDOWN` | 600 | кулдаун между вспышками |
| `JEALOUSY_DIRECT_REPLY_HINT_TTL` | 120 | подсказка после возвращения любимчика |
| `JEALOUSY_RECENT_PUBLIC_LIMIT` | 5 | сколько последних контекстов хранить |
| `JEALOUSY_AFTERGLOW_MIN` | 300 | минимум afterglow |
| `JEALOUSY_AFTERGLOW_MAX` | 600 | максимум afterglow |
| `JEALOUSY_POST_RETURN_STRIKES` | 3 | сколько ответов остаются колкими |
| `JEALOUSY_SOFT_RESET_DECREMENT` | 2 | насколько снижается счётчик при прямом обращении |

## 13. Дневник планеты и вердикт дня

### Дневник планеты

`analysis.py` периодически создаёт записи в `planet_diary` по сообщениям общего чата.

Настройки:

```python
DIARY_INTERVAL = 7200
DIARY_MIN_MESSAGES = 40
```

В текущем `main.py` запуск ориентируется на количество новых сообщений после последней записи: если набралось минимум `DIARY_MIN_MESSAGES`, запускается `_generate_diary_background()`.

### Команда `!дневник`

Админская команда. Берёт все записи дневника за текущие локальные сутки и делает цельную сводку до ~800 символов.

Защиты:

- не отдавать сырой дамп записей;
- не пропускать системный промпт в ответ;
- перегенерировать слишком длинный ответ;
- fallback-сводка, если LLM не справилась.

### Ежедневный вердикт

`daily_verdict.py` после 22:00 генерирует публичный вердикт за день.

- дата берётся через `datetime.now().strftime("%Y-%m-%d")`;
- сообщения дня берутся через `get_global_messages_for_date(report_date, limit=180)`;
- результат сохраняется в `daily_reports`;
- за одну дату вердикт пишется только один раз.

## 14. Настроения, активность и приветствия

### Настроения

Текущее настроение выбирается случайно при запуске и меняется раз в `MOOD_CHANGE_INTERVAL`.

Админ может сменить настроение командой:

```text
!настроение N
```

### Активность

`state.update_activity()` выбирает текущую активность по времени суток из `DAY_ACTIVITY_SCHEDULE`. Активность попадает в промпт и в генерацию хуков при тишине.

### Приветствия

Когда пользователь входит на планету, бот с вероятностью `0.25` ставит в очередь `greeting`.

Приветствия стараются:

- не повторяться дословно;
- иногда шутить над ником;
- отдельно обрабатывать дефолтные ники вида `user123`.

### Хуки при тишине

Если чат молчит дольше `HOOK_SILENCE`, а последний хук был не раньше `HOOK_COOLDOWN`, бот генерирует короткое сообщение для оживления чата.

### Боковые комментарии в активный разговор

`group_comment` — это отдельная механика редкого встревания в чужой активный диалог. Она отличается от хуков при тишине: хук нужен, когда чат молчит, а `group_comment` срабатывает только когда люди уже разговаривают.

В `main.py` задача ставится в очередь только если выполнены условия:

- есть свежие осмысленные сообщения;
- очередь ответов пуста;
- в последних сообщениях есть мини-диалог: минимум 4 сообщения и минимум 2 автора по последним 8 сообщениям;
- прошёл локальный кулдаун `GROUP_COMMENT_COOLDOWN = 180`;
- сработал шанс `random.uniform(0.08, 0.12)`.

В job передаётся `recent_messages` — последние сообщения общего чата. Это важно: воркер больше не пытается сам угадать контекст по пустому query, а реагирует именно на тот фрагмент, из-за которого комментарий был поставлен в очередь.

В `worker.py` ветка `group_comment`:

1. форматирует свежий контекст через `_format_group_comment_context(...)`;
2. определяет тон через `_detect_group_comment_tone(...)`;
3. пропускает шутливый комментарий при серьёзных self-harm маркерах;
4. выбирает инструкцию из `_GROUP_COMMENT_TONE_INSTRUCTIONS`;
5. вызывает модель с отдельным коротким system prompt;
6. чистит результат через `_clean_group_comment_line(...)`;
7. если результат не подходит, берёт безопасный fallback из `_fallback_group_comment(...)`.

Поддерживаемые тона:

| Тон | Когда срабатывает | Как отвечает |
|---|---|---|
| `dark_joke` | смерть, морг, гроб, кладбище, трупы, чёрный юмор | сухо, язвительно, без сердечек и романтики |
| `conflict` | спор, подколы, раздражение | как ехидный наблюдатель |
| `flirt` | флирт, сердечки, поцелуи, нежности | колко и ревниво-насмешливо, без explicit |
| `absurd` | мемы, хаос, кринж, абсурд | устало-смешливая реакция на цирк |
| `neutral` | обычный разговор | короткий наблюдательный сарказм |
| `skip` | серьёзные self-harm сигналы | комментарий не отправляется |

Главное правило для этой механики:

```text
контекст сцены важнее текущего настроения бота
```

Поэтому если чат шутит про внезапную смерть, бот должен отвечать в `dark_joke`-тоне, а не уходить в нежные фразы, сердечки и «лепестки под дождём».

## 17. Как создать другого персонажа

Минимальный набор изменений:

1. В `config.py` поменять:
   - `CHARACTER_NAME`;
   - `LOG_NAME`;
   - `NAME_TRIGGERS`;
   - `NAME_WORD_FORMS`;
   - `SYSTEM_PROMPTS`;
   - `MODE_NAMES`.
2. Обновить шаблоны:
   - `FALLBACK_RESPONSE`;
   - `GREETING_FALLBACK_TEMPLATE`;
   - `SEARCH_ERROR_TEMPLATE`;
   - `INJECTION_RESPONSES`;
   - `HOOKS` и `MOOD_HOOKS`.
3. Проверить `GALAXY_TERMINOLOGY`, если персонаж остаётся в Galaxy.
4. Проверить ограничения флирта и NSFW в системном промпте.
5. Проверить ревность: подходит ли логика «любимчик/соперник» новому персонажу.
6. Если меняется стиль речи, обновить `STYLE_HINTS` и анти-повторные паттерны.

## 18. Что важно при доработках

### Не ломать очередь

Для LLM-ответов лучше использовать:

```python
await state.enqueue_llm_job(username, text, current_mood)
```

а не прямой `put({"type": "llm" ...})`, если нужно сохранить склейку сообщений.

### Не вызывать модель напрямую из `main.py` без причины

Обычные ответы должны идти через очередь и воркер, иначе легко сломать кулдаун, порядок сообщений и достижения.

Исключение — хук при тишине, который сейчас генерируется прямо в `main.py`. Боковые комментарии `group_comment` идут через очередь и обрабатываются в `worker.py`.

### Не подмешивать web search вместо system prompt

Поиск должен работать так:

```text
system = полноценный промпт персонажа
user = инструкция найти информацию
```

Результат поиска не должен заменять системный промпт. Это уже исправлено в `responder.py`.

### Не давать LLM трогать профиль без фильтра

Новые поля профиля должны проходить проверку безопасности, иначе пользователь может записать prompt injection в `display_name`, `city` или другое поле.

### Не использовать память как абсолютную истину

Факты и ярлыки могут быть ошибочными. Профиль имеет больший приоритет, но даже его лучше обновлять только из прямых самоописаний или валидированных фактов.

### Не превращать погоду и науку в справочник

Актуальная задумка поведения:

- культура/музыка/мемы/фильмы — можно искать;
- новости/цены/расписания — можно искать;
- погода — не искать, отшутиться;
- сложная школа/наука — не искать, отвечать как персонаж.

### Не забывать про SELF_IDENTITY

Если сообщение выглядит как «Маша, кто такая Анька?», нельзя отдавать его в обычный gossip-flow. Иначе модель может ответить про Машу или другого человека из контекста.

### Не расширять `group_comment` до полноценного ответа

`group_comment` должен оставаться короткой боковой репликой без ника в начале. Для него нельзя использовать общий тяжёлый `get_system_prompt()`: там слишком много личной памяти, настроения и правил обычного диалога. Эта механика должна опираться на свежий фрагмент общего чата и тон сцены.

Особенно важно для `dark_joke`: при чёрном юморе запрещены романтичные и нежные образы, сердечки, сочувственные открытки и философская лирика. Иначе бот снова начнёт отвечать не в тон разговора.

### Не хранить секреты в документации

В `config.py` может лежать реальный API-ключ. В документации всегда использовать `xai-...`.
